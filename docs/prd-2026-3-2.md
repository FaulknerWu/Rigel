## 执行摘要

Rigel 是一个面向 AI Coding Agent 的可解释代码语义检索系统。它对 Python 代码仓库进行离线索引，构建实体语义图、层次化摘要与混合检索索引，并以 MCP 服务器的形式向 AI Coding Agent 暴露能力。其核心职责是对代码客观信息进行整理、归纳、引用与结构化呈现，输出事实、证据、链路与上下文，帮助 Agent 在大型代码仓库中更快、更准地定位逻辑、追踪关系。

## 产品目标

### 业务目标

- 建立一个可被 AI Coding Agent 直接消费代码上下文基础设施。
- 显著降低 Agent 在中大型代码仓库中的上下文理解成本
- 让语义检索与查询结果具备结构化证据、来源链路与上下文，提升 Agent 分析可靠性
- 以本地优先、增量更新的模式满足真实开发流程中的使用效率要求
- 为后续多语言支持与更深层代码分析能力打下基础

### 成功指标

- 在目标 Python 仓库中，索引成功率达到或超过 95%
- 自然语言查询的 Top-5 结果中，至少包含 1 条强相关结果
- 每条检索结果都能返回明确命中原因、证据位置与来源链路
- 调用关系与依赖关系结果中，确定性关系与候选关系必须 100% 分层展示

## 功能需求

### Epic 1：索引构建引擎

#### FR-001: 项目基建与数据框架

**优先级：** 必须有

**描述：**
系统搭建基础的数据存储、配置管理框架，为后续索引、检索、回答提供统一的技术地基。

**验收标准：**

- [ ] 初始化数据库
- [ ] 实现基础 CRUD 接口
- [ ] 建立配置管理系统，至少覆盖：目标仓库路径、数据存储路径、LLM/Embedding 模型标识与连接参数
- [ ] 支持注册目标仓库，生成唯一 repo_id，记录仓库根路径与元信息
- [ ] 数据库 Schema 变更时支持重建

**依赖：** 无

---

#### FR-002: Python AST 解析与实体抽取

**优先级：** 必须有

**描述：**
系统基于 AST 解析 Python 源码（Tree-Sitter官方原生支持），抽取文件、类、函数等核心实体作为节点。

**验收标准：**

- [ ] 至少支持文件、类、函数三类基础实体
- [ ] method 作为函数的子类标识，用于区分类方法与顶层函数
- [ ] 每个实体都包含足够定位与区分的元信息（名称、全限定名、文件路径、模块路径、起止行号等）
- [ ] 遇到单文件解析失败时，不阻断整个仓库索引流程
- [ ] 不支持的语法模式记入警告列表

**依赖：** FR-001

---

#### FR-003: 结构关系构建

**优先级：** 必须有

**描述：**
系统在索引阶段建立实体间的结构关系，用于表达代码中的定义归属、调用依赖、导入依赖、继承扩展与接口实现等语义联系。

**验收标准：**

- [ ] 至少支持定义（defines）、调用（calls）、导入（imports）、继承（inherits）、实现（implements）五种关系；其中定义表达文件、类、函数等作用域对下级实体的定义归属；导入表达跨文件/跨模块的依赖引用
- [ ] implements 必须区分两种置信度：显式继承 ABC/Protocol 的为确定性（deterministic）；仅结构化满足协议签名的为候选（candidate）
- [ ] 每条关系记录必须可追溯到源与目标节点，并携带关系类型、置信度与证据位置
- [ ] 同一实体在不同文件中的引用可被统一归档；跨文件引用归档需要解析器/索引器层支撑，不能仅靠单文件 AST

**依赖：** FR-002

---

#### FR-004: 增量索引

**优先级：** 必须有

**描述：**
系统支持仅对发生变更的文件执行增量更新，避免每次都全量重建。

**验收标准：**

- [ ] 可检测新增、修改、删除文件
- [ ] 增量索引仅重建受影响实体与关系
- [ ] 增量更新的影响范围不限于"变更文件"本身，还必须扩展到：受影响实体、反向关系（引用了变更实体的其他实体）、以及依赖这些实体的摘要与向量
- [ ] 删除文件后，历史实体与关系会被清理
- [ ] 返回本次增量更新影响范围统计

**依赖：** FR-001, FR-003

---

#### FR-005: 层次化语义摘要与向量表示生成

**优先级：** 必须有

**描述：**
系统通过 LLM 为文件、类、函数等实体生成可检索的语义表示，作为节点索引。该表示由两部分组成：

1. 源码锚定摘要：可回溯、忠实于源码事实的摘要文本，强制携带源码锚点；
2. 基于实体语义表示生成的 embedding 向量，用于自然语言语义召回。

**验收标准：**

- [ ] 至少为文件、类（含 protocol/ABC）、函数（含 method）三类实体生成语义表示；物理存储方式不做强制约束
- [ ] 每个实体都持久化摘要文本、embedding 向量及生成元信息
- [ ] 摘要必须锚定源码事实，描述职责、输入输出与核心行为，并强制携带源码锚点（文件路径 + 行号范围），支撑后续引用验证
- [ ] 增量索引时仅重算受影响实体的摘要与向量
- [ ] 摘要或向量生成失败时不影响结构索引落库，但必须写入警告与失败原因

**依赖：** FR-002

### Epic 2：检索与证据输出

#### FR-006: 混合召回、子图构建与统一重排

**优先级：** 必须有

**描述：**
系统通过 LLM 将用户的自然语言问题转换为标准的查询语句（具体释义见下文），并通过向量查询从索引中召回相关种子节点，沿关系对种子节点执行多跳扩展，构建与查询语义相关的子图。多个种子节点对应多个子图，需要将重叠部分进行合并。

如何转换：这一步的最终目标是让用户问题更“可检索”。当原问题里有**省略、指代、上下文依赖、模糊表达、决策类问题**时，把它改写成信息完整的查询，通常更利于后续检索器工作。比如用户说“它去年那个政策后来怎么改了”，这里的“它”“那个政策”都不完整；LLM 通过对话的上下文改成“某公司 2025 年隐私政策在后续版本中有哪些修改”，检索目标就更明确，且结果有利于 LLM 分析。

目前所规划的具体方案如下：

- 消除代词和上下文省略
- 把口语问题改成自包含问题

- 补上领域里的同义词、别名、标准术语

- 用 LLM 把复杂问题拆成更聚焦的子查询

- 对原问题做扩展，增加召回面
