## 执行摘要

Rigel 是一个面向 AI Coding Agent 的可解释代码语义检索引擎。它通过对 Python 代码仓库进行离线索引，构建实体级语义图、层次化摘要与混合检索索引，并以 MCP Server 的形式向 AI Coding Agent 暴露结构化能力。它负责返回对代码客观信息的整理、归纳、引用与结构化呈现，输出事实、证据、链路与上下文，帮助 Agent 在大型代码仓库中更快、更准地定位逻辑、追踪关系，并形成可回溯的信息底座。

## 产品目标

### 业务目标

- 建立一个可被 AI Coding Agent 直接消费的代码语义检索与查询基础设施
- 显著降低 Agent 在中大型代码仓库中的上下文理解成本
- 让语义检索与查询结果具备结构化证据、来源链路与上下文，提升 Agent 分析可靠性
- 以本地优先、增量更新的模式满足真实开发流程中的使用效率要求
- 为后续多语言支持与更深层代码分析能力打下基础

### 成功指标

- 在目标 Python 仓库中，索引成功率达到或超过 95%
- 自然语言查询的 Top-5 结果中，至少包含 1 条强相关结果
- 每条检索结果都能返回明确命中原因、证据位置与来源链路
- 调用关系与依赖关系结果中，确定性关系与候选关系必须 100% 分层展示

### Epic 1：索引构建引擎

#### FR-001: Python 仓库索引构建

**优先级：** 必须有

**描述：**
系统提供仓库索引能力，将指定 Python 仓库构建为可查询的本地索引。

**验收标准：**

- [ ] 支持输入仓库路径触发索引构建
- [ ] 支持显式触发全量重建
- [ ] 返回文件数、实体数、关系数、耗时与警告信息
- [ ] 索引过程失败时返回可读错误，而不是静默失败

**依赖：** 无

---

#### FR-002: Python AST 解析与实体抽取

**优先级：** 必须有

**描述：**
系统基于 AST 解析 Python 源码，抽取文件、类、函数、接口等核心实体作为节点，并为后续扩展打下基础。

**验收标准：**

- [ ] 至少支持文件（file）、类（class）、函数（function）、接口（interface）四类基础实体
- [ ] 每个实体都包含名称、文件路径、起止行号等信息
- [ ] 遇到单文件解析失败时，不阻断整个仓库索引流程
- [ ] 无法支持的语法模式会进入警告列表

**依赖：** FR-001

---

#### FR-003: 结构关系构建

**优先级：** 必须有

**描述：**
系统在索引阶段建立实体间的结构关系，用于表达代码中的定义归属、调用依赖、继承扩展与接口实现等语义联系。

**验收标准：**

- [ ] 至少支持定义（defines）、调用（calls）、继承（inherits）、实现（implements）四种关系，其中定义表达文件、类、函数等作用域对下级实体的定义归属，例如文件定义类、类定义方法、函数定义内部函数
- [ ] 关系记录包含源节点、目标节点、关系类型与证据位置
- [ ] 同一实体在不同文件中的引用可被统一归档
- [ ] 基于 AST 的确定性关系与候选关系不混淆

**依赖：** FR-002

---

#### FR-004: 增量索引

**优先级：** 必须有

**描述：**
系统支持仅对发生变更的文件执行增量更新，避免每次都全量重建。

**验收标准：**

- [ ] 可检测新增、修改、删除文件
- [ ] 增量索引仅重建受影响实体与关系
- [ ] 删除文件后，历史实体与关系会被清理
- [ ] 返回本次增量更新影响范围统计

**依赖：** FR-001, FR-003

---

#### FR-005: 层次化语义摘要与向量表示生成

**优先级：** 必须有

**描述：**
系统通过 LLM 为文件、类、函数、接口等实体生成可检索的语义表示。该表示由两部分组成：

1. 可回溯、忠实于源码事实的摘要文本；
2. 基于实体语义表示生成的 embedding 向量，用于自然语言语义召回。

**验收标准：**

- [ ] 至少为 file、class、function、interface 四类实体生成语义表示及衍生数据，并直接储存到对应节点
- [ ] 每个实体都落库摘要文本、embedding 向量、所使用的 embedding 模型、生成时间、源码锚点
- [ ] 摘要内容描述职责、输入输出与核心行为
- [ ] 增量索引时仅重算受影响实体的摘要与向量
- [ ] 摘要或向量生成失败时不影响结构索引落库，但必须写入警告与失败原因

**依赖：** FR-002

---

### Epic 2：检索与证据输出

#### FR-006: 混合召回、子图构建与统一重排

**优先级：** 必须有

**描述：**
系统将自然语言查询转换为统一查询表示（当前方案通过 LLM 实现），从实体语义向量索引中召回相关种子节点，并沿关系对种子节点执行多跳扩展，构建与查询语义相关的子图。

**验收标准：**

- [ ] 支持将查询转换为查询向量，并保留路径过滤、节点类型过滤等结构化约束
- [ ] 基于实体语义向量索引召回 Top-K 种子节点（可考虑引入 Reranker 模型）
- [ ] 支持沿 defines、calls、inherits、implements 四类关系执行受控扩展；扩展深度、最大分支数与超时均可配置
- [ ] 当同一节点经多条路径命中时，能够聚合展示，并保留各自路径与证据
- [ ] 输出结果包含 nodes、edges、paths 三类结构

**依赖：** FR-003, FR-005

---
